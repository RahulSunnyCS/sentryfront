## Scan-Submit Critical Path (non-hermetic — real backend)

🔴 Submitting a URL fires a real POST to /api/v1/scans with the correct JSON body
  Scenario    : On /en, start page.waitForRequest('**/api/v1/scans') BEFORE submitting. Fill [data-testid=hero-url-input] with https://example.com. Click [data-testid=hero-scan-submit]. Await the captured request.
  Pass if     : The captured request method is POST and its JSON body matches { url: 'https://example.com' } (toMatchObject semantics — extra keys are allowed).
  Fail if     : No matching request is observed within the timeout; or the request method is not POST; or the body is missing, malformed, or contains a URL other than https://example.com.
  Automatable : yes
  @critical

🔴 After form submission the browser navigates to the scan result URL
  Scenario    : After the POST from the critical-path test above fires and the server returns 201, wait for the browser URL to change.
  Pass if     : Browser URL matches the pattern /\/en\/scan\/[^/?]+/ within the configured navigation timeout (query string may be present and is not asserted).
  Fail if     : URL does not transition to /en/scan/<real-id> within the timeout, or a network or JS error prevents navigation.
  Automatable : yes
  @critical

🟡 Submitting with an empty input sends example.com as the URL and navigates to the scan result page
  Scenario    : On /en, clear [data-testid=hero-url-input] so it is empty. Start page.waitForRequest('**/api/v1/scans'). Click [data-testid=hero-scan-submit]. Await the captured request and then wait for URL transition.
  Pass if     : Captured POST body contains { url: 'example.com' } and the browser URL transitions to /\/en\/scan\/[^/?]+/.
  Fail if     : POST body contains an empty, null, or different URL; or the URL transition does not occur within the timeout.
  Automatable : yes
  @functional

  Fail if     : Vitest attempts to run any file under e2e/, producing import errors, test failures, or a non-zero exit code attributable to e2e specs.
  Automatable : yes
  @critical

🔴 The test:e2e npm script exists and is wired to the Playwright CLI
  Scenario    : Inspect package.json scripts for a test:e2e key. Confirm its value resolves to playwright test.
  Pass if     : test:e2e is present in package.json and its value is playwright test (or npx playwright test).
  Fail if     : test:e2e is absent from package.json, or it resolves to a different command.
  Automatable : yes
  @critical

  Pass if     : test:e2e:ui is present and its value contains --ui.
  Fail if     : Key is absent.
  Automatable : yes
  @functional

🔴 playwright.config.ts exists at the repo root with correct testDir and baseURL
  Scenario    : Read playwright.config.ts from the repo root. Assert testDir is './e2e' and baseURL is 'http://localhost:3000'.

## CI — GitHub Actions e2e Job

🔴 The CI e2e job is defined in .github/workflows/test.yml
  Scenario    : Read .github/workflows/test.yml and locate a job named e2e (or logically equivalent). Confirm it includes npm run test:e2e as a step.
  Pass if     : A distinct e2e job exists; npm run test:e2e appears as a step.
  Fail if     : No e2e job is present, or only the existing test and build jobs are defined.
  Automatable : yes
  @critical

🔴 The CI e2e job does NOT set continue-on-error
  Scenario    : Inspect the e2e job definition in test.yml for a continue-on-error field.
  Pass if     : continue-on-error is absent or explicitly false — a failing @critical E2E test must turn the CI workflow RED, enforcing the regression gate in CI.
  Fail if     : continue-on-error is true or equivalent, which would silently swallow critical test failures in CI.
  Automatable : yes
  @critical

🟡 The CI e2e job uses Node 20.19 and runs npm ci before test execution
  Scenario    : Inspect the e2e job steps. Check the node-version setup action and the presence of npm ci before npm run test:e2e.
  Pass if     : node-version is 20.19 (or 20.x equivalent); npm ci is a step preceding npm run test:e2e.
  Fail if     : A different Node major version is used, or npm ci is absent.
  Automatable : yes
  @functional

🟡 The CI e2e job caches the Playwright browser binary keyed on the Playwright version
  Scenario    : Inspect the e2e job for a cache step targeting ~/.cache/ms-playwright with a cache key that includes the Playwright version string.
  Pass if     : A cache step is present with the correct path and a version-derived cache key.
  Fail if     : No cache step is present (CI will be slow and re-downloads browsers on every run).
  Automatable : yes
  @functional

🟡 The CI e2e job installs Chromium with system dependencies
  Scenario    : Inspect the e2e job steps for a Playwright install step.
  Pass if     : Step runs npx playwright install --with-deps chromium (or equivalent for Chromium only).
  Fail if     : Step is absent, installs all browsers (wastes CI minutes), or omits --with-deps (binary may fail on the CI runner).
  Automatable : yes
  @functional

🟡 The CI e2e job uploads the playwright-report/ artifact unconditionally
  Scenario    : Inspect the e2e job for an upload-artifact step referencing playwright-report/ with if: always().
  Pass if     : Step is present, path includes playwright-report/, and the condition is always().
  Fail if     : Step is absent or the condition is omitted, meaning failure reports are not uploaded for debugging.
  Automatable : yes
  @functional

🟡 The existing test and build CI jobs are unchanged after the e2e job addition
  Scenario    : Read the test and build job definitions in test.yml and verify their steps, Node versions, and triggers are unmodified.
  Pass if     : No modifications to any existing job step or setting compared to the pre-change baseline.
  Fail if     : Any existing job step, Node version, or trigger has been altered.
  Automatable : partial
