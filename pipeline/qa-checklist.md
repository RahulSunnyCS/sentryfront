# QA Checklist

Generated by: qa-planner
Plan context: Provably-complete Playwright E2E coverage for every VibeSafe page (~28) and component (31) at risk-tiered depth, plus a Wave-0 prod-guard vitest regression and a Wave-1 auth-seed probe.
Risk level: HIGH
Auth/PII escalated: yes (risk_flags include auth AND payment/billing — Opus, high effort)

Notes for the E2E Test Writer (Phase 5) and Automation Gate (Phase 6):
- This artifact is machine-readable and consumed unchanged. Each case carries exactly one tier tag: @critical / @functional / @non-blocker.
- Wave-0 prod-guard is a Vitest unit/integration case (Automatable: no for Playwright — implement in vitest, not e2e/). It still blocks Gate 2 as @critical.
- Wave-1 PROBE is a precondition gate: if it fails with PrismaClientInitializationError it is a HARD Wave-1 STOP (not a normal DIRECT failure).
- Pricing: do NOT assert dollar amounts (May-2026 pivot in flux). Assert credit/subscription state transitions and tier behaviour only. Never hit real Stripe — only the PAYMENT_TEST_FLOW bypass path.
- Tier-isolation: every tier-sensitive/tier-mutating spec seeds its OWN unique-email user and cleans up in afterAll. Never mutate the global seeded user.
- Scan-lifecycle states are pre-seeded Scan/Finding/ScanEvent rows. Only ONE spec calls the real /api/v1/scans and asserts submission+redirect only (never awaits a terminal state).

## Wave 0 — Production Bypass Guard (Vitest, blocking)

🔴 In production with the payment test bypass enabled, the checkout route refuses before doing any work
  Scenario    : Set NODE_ENV='production' and PAYMENT_TEST_FLOW='true'. POST to the checkout route handler with a valid-looking request.
  Pass if     : Response status is 404; getCurrentUser is never called; prisma.user.update is never called; the response body contains no tier value and no session payload; a guard log line is emitted.
  Fail if     : Status is anything other than 404, OR getCurrentUser is invoked, OR prisma.user.update is invoked, OR the body leaks a tier/session, OR the route proceeds past the guard.
  Automatable : no (Vitest unit/integration — not Playwright; spy on getCurrentUser and prisma.user.update)
  @critical

🔴 The production guard returns before authentication so it cannot be probed for existence
  Scenario    : Same prod+flag setup, but send an unauthenticated request (no session cookie).
  Pass if     : 404 returned identically to the not-enabled 404 at the route's existing branch; no auth attempted; behaviour is indistinguishable from "feature not enabled" (no bypass existence leak).
  Fail if     : A different status/body for unauth vs authed reveals the bypass exists, OR auth runs before the guard.
  Automatable : no (Vitest)
  @critical

🟡 With the bypass enabled in development the checkout route still functions
  Scenario    : NODE_ENV='development', PAYMENT_TEST_FLOW='true', authenticated user, valid price selection.
  Pass if     : The guard does NOT trigger (404 not returned for the guard reason); the dev bypass path proceeds normally.
  Fail if     : The prod guard fires in development and blocks the legitimate dev bypass.
  Automatable : no (Vitest)
  @functional

🟢 The guard log line does not contain user-identifying data
  Scenario    : Inspect the guard log emitted in the prod+flag case.
  Pass if     : Log records the refusal reason without email, tier, session token, or request body.
  Automatable : no (Vitest)
  @non-blocker

## Wave 1 — Auth-Seed Probe & Shared Infra (blocking precondition)

🔴 The server reports the NextAuth provider, not the Supabase stub
  Scenario    : After full cold compile, query the health/feature-status surface that exposes getAuthProvider().
  Pass if     : getAuthProvider() === 'nextauth' is reported by the server.
  Fail if     : Provider is 'supabase' or unset (which would silently return no user and make every authed spec a false negative).
  Automatable : yes
  @critical

🔴 A seeded DB session resolves to a real user through a protected route
  Scenario    : Seed a User + Session row via auth-seed (cookie next-auth.session-token == Session.sessionToken, expires now+30d). Request a real protected route with that cookie.
  Pass if     : getCurrentUser() resolves to the seeded user (route returns the authenticated 2xx view, not a redirect to login).
  Fail if     : The route redirects to login or returns unauthenticated, indicating the DB-session contract is broken.
  Automatable : yes
  @critical

🔴 The probe performs a real server-side DB write and read-back
  Scenario    : Through the running server, trigger a path that writes a row and then reads it back.
  Pass if     : Write+read succeed end to end through the server.
  Fail if     : Any non-2xx. A PrismaClientInitializationError (error class, not message substring) is a HARD Wave-1 STOP — surface immediately, no retry. Any other non-2xx is a normal DIRECT Wave-1 failure.
  Automatable : yes
  @critical

🟡 globalSetup leaves the schema datasource on sqlite
  Scenario    : After globalSetup runs (delete e2e.db*, db-config development, db push, seed), inspect prisma/schema.prisma datasource provider.
  Pass if     : provider == sqlite.
  Fail if     : provider == postgresql (hard fail — the seed/probe would target the wrong DB).
  Automatable : partial (assertion inside globalSetup; surfaced as a probe-spec check)
  @functional

🟢 The expired-session cookie does not authenticate
  Scenario    : Seed a Session row with expires in the past; request a protected route with that cookie.
  Pass if     : Treated as unauthenticated (redirect to login).
  Automatable : yes
  @non-blocker
