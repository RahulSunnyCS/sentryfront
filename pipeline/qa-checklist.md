## Legal Honesty and Disclaimer Constraints

🔴 The compliance section never uses the word "compliant", "certified", "attested", or "passed" for any framework
  Scenario    : Scan any URL with complianceScanning enabled; inspect every rendered string in the Compliance section of the report for attestation language. Also inspect RawFinding.title, .explanation, and .impact fields for P5-01 through P5-06 findings.
  Pass if     : No finding title, explanation, UI heading, or tooltip contains language that asserts conformance or attestation to any regulation (GDPR, CCPA, PCI-DSS, or equivalent).
  Fail if     : Any string visible to a user states or strongly implies the scanned site is "GDPR compliant", "CCPA certified", "passes", or similar. Any finding severity label is misused to convey a legal verdict.
  Automatable : yes
  @critical

🔴 The disclaimer is present and prominently rendered on every report that shows a compliance signal score or framework summary
  Scenario    : Load a report page for a completed scan that has at least one P5 finding. Inspect the rendered DOM of the Compliance section. Also test the case where all P5 modules emit zero findings.
  Pass if     : A disclaimer stating the signals are not legal advice and do not constitute an attestation or audit is visible in the Compliance section in both cases (findings present and findings absent).
  Fail if     : The disclaimer is absent, hidden behind an interaction, or only shown conditionally based on finding count. The disclaimer must not be removable by a feature flag without removing the entire section.
  Automatable : yes
  @critical

🔴 Findings emitted by P5-01 through P5-06 state observable facts only, not conclusions about regulatory compliance
  Scenario    : Trigger each P5 module individually (unit-level) against crafted CrawlResult fixtures: one with a clear signal present (e.g. OneTrust script detected), one with no signal. Inspect every field of each returned RawFinding.
  Pass if     : Every finding title and explanation describes what was observed (e.g. "No cookie consent mechanism detected in DOM") rather than what it means legally (e.g. "Site is not GDPR compliant"). Regulation names are permitted as context only (e.g. "Commonly required under GDPR-type frameworks").
  Fail if     : Any RawFinding field contains a definitive legal conclusion. Any finding uses "violation", "non-compliant", "failed [regulation]", or equivalent as a verdict.
  Automatable : yes
  @critical

🔴 The framework-signal score label and display wording make its non-binding nature unambiguous
  Scenario    : Render a report with a complianceSignalScore of 100 (all signals observed). Inspect the UI label and any tooltip for the score.
  Pass if     : The score is labelled as a "signal coverage" percentage or equivalent, not a compliance score. A score of 100 does not render as "fully compliant". The non-binding label is co-located with the score, not just in the footer.
  Fail if     : A perfect score implies full compliance in the UI. The score label uses the word "compliance" without a qualifying modifier ("signal", "indicator", "heuristic", or similar).
  Automatable : yes
  @critical

## Flag-Off (complianceScanning Disabled) — Byte-Identical Output

🔴 When complianceScanning feature flag is false, runScanner returns a ScannerResult with no P5 findings and no compliance-specific fields
  Scenario    : Set features.complianceScanning = false. Call runScanner() on a target URL that would normally trigger multiple P5 findings. Serialize the returned ScannerResult and compare its shape against a pre-feature baseline.
  Pass if     : The result contains no complianceSignalScore, no complianceFrameworkSummary, no P5-Compliance key in moduleFindingCounts, and zero findings with a moduleId of P5-xx. The shape is structurally identical to what runScanner returned before the P5 group was added.
  Fail if     : Any compliance field or P5 finding appears in the result when the flag is off. moduleFindingCounts gains a P5-Compliance key with value 0 (a zero-count key is still a shape change — the key must be absent entirely).
  Automatable : yes
  @critical

🔴 When complianceScanning is false, the report UI renders the original CompliancePlaceholder (or nothing), not the new compliance component
  Scenario    : Render a report page with a scan result that has no P5 fields (flag-off shape). Inspect the Compliance section of the report DOM.
  Pass if     : The Compliance section either renders the original CompliancePlaceholder ("coming soon") or is absent. The new compliance component with framework summary and disclaimer is not rendered.
  Fail if     : The new compliance component appears when no P5 data is present, or the section throws a runtime error due to missing optional fields.
  Automatable : yes
  @critical

🟡 scan-worker correctly omits all compliance fields from the persisted Scan record when complianceScanning is false
  Scenario    : Run a full scan end-to-end with the flag off. Inspect the database record via Prisma for the resulting Scan row.
  Pass if     : No compliance-related columns or JSON fields are written to the Scan record that were not present before the P5 feature existed.
  Fail if     : Null or empty compliance columns appear in the persisted record that differ from the pre-feature schema shape.
  Automatable : partial
  @functional

## No New Outbound Requests

🔴 None of the P5-01 through P5-06 modules make network calls; they operate exclusively on the already-populated CrawlResult
  Scenario    : Run each P5 module in isolation with a valid CrawlResult fixture. Intercept all outbound HTTP calls during module execution (mock fetch and any http/https node modules at test boundary).
  Pass if     : Zero outbound requests are made by any P5 module during execution. Modules only read properties of the CrawlResult argument.
  Fail if     : Any P5 module calls fetch, http.get, https.get, axios, or any equivalent; or accesses the network through a shared utility.
  Automatable : yes
  @critical

🟡 The compliance orchestrator (compliance.ts) does not introduce any async I/O that is not present in the CrawlResult
  Scenario    : Review the compliance.ts wrapper and each P5 module signature for any async operation that could resolve to a network call. Run with a completely offline CrawlResult fixture.
  Pass if     : All five P5 modules and the orchestrator are synchronous or only await in-process operations (e.g. cheerio parse of already-fetched HTML).
  Fail if     : Any await in a P5 module or the orchestrator resolves to an external resource, even conditionally.
  Automatable : yes
  @functional

## Scan Timeout Budget

🔴 The full P5 compliance group (all six modules) completes within 5 seconds on a realistic large CrawlResult
  Scenario    : Construct a worst-case CrawlResult fixture: 500KB HTML with 50 third-party script URLs, 30 cookies, and a full networkRequests array of 200 entries. Run runComplianceModules() and measure wall-clock time.
  Pass if     : The compliance group finishes in under 5 seconds. The overall runScanner() call does not push a scan past the 120-second hard timeout in any scenario where P1-P4 groups are all passing within their own budgets.
  Fail if     : Any single P5 module exceeds 2 seconds on the large fixture, or the group total exceeds 5 seconds. Any P5 module introduces unbounded iteration over networkRequests or cookies without a size cap.
  Automatable : yes
  @critical

🟡 The compliance group runs after the accessibility result is available and the execution order in scanner/index.ts is correct
  Scenario    : Inspect the ordering of awaits/Promise.all in scanner/index.ts. Trace the accessibilityScore threading from runAccessibilityModules through to the compliance orchestrator context argument.
  Pass if     : The P5 group executes in a position where accessibilityResult is non-null (when features.accessibilityScanning is also true). The accessibilityScore context argument is the integer 0-100 from accessibilityResult.accessibilityScore.
  Fail if     : The compliance group runs before or in parallel with the accessibility group, resulting in accessibilityScore being undefined when P5-04 WCAG Attestation executes. Or the compliance group runs unconditionally before the accessibility feature-flag check.
  Automatable : yes
  @functional

🟢 When accessibilityScanning is disabled but complianceScanning is enabled, P5-04 degrades gracefully
  Scenario    : Set features.accessibilityScanning = false and features.complianceScanning = true. Run a scan. Inspect the P5-04 findings.
  Pass if     : P5-04 either emits an INFO finding noting the Lighthouse a11y score was unavailable, or emits no finding rather than throwing. The scan does not error.
  Automatable : yes
  @non-blocker

## ScannerResult and Persisted Shape Back-Compatibility

🔴 Old persisted scan records (pre-P5) deserialize correctly in the report-view without errors when complianceSignalScore and complianceFrameworkSummary are absent
  Scenario    : Construct a ScannerResult fixture with none of the P5 fields (simulating a scan run before the feature shipped). Pass it to the report-view compliance section renderer. Also test with a null complianceSignalScore.
  Pass if     : The report renders without throwing. The compliance section shows the placeholder or a graceful "no compliance data" state. No "cannot read properties of undefined" errors reach the error boundary.
  Fail if     : The report crashes or shows an unhandled exception when P5 fields are absent. Optional fields are accessed without null guards.
  Automatable : yes
  @critical

🔴 The ScannerResult type additions for P5 are all optional fields and TypeScript strict-mode compilation passes
  Scenario    : Run tsc --noEmit (or npm run build which includes type checking) after the P5 type additions are applied to scanner/index.ts's ScannerResult interface.
  Pass if     : Zero TypeScript errors related to the new P5 fields. Existing callers of ScannerResult that do not read P5 fields compile without changes.
  Fail if     : Any existing file requires modification purely to satisfy the new required fields. Any P5 field is added as a required (non-optional) property.
  Automatable : yes
  @critical

🟡 scan-worker correctly maps complianceSignalScore and complianceFrameworkSummary from ScannerResult to the Scan persistence layer when the flag is on
  Scenario    : Run a full scan end-to-end with complianceScanning enabled. Verify the compliance fields are persisted and round-trip through the report API to the report-view.
  Pass if     : complianceSignalScore and complianceFrameworkSummary appear in the API response for the scan and are rendered in the UI.
  Fail if     : The fields are computed in runScanner but silently dropped by scan-worker before persistence, causing the report to always show 0 signals.
  Automatable : partial
  @functional

🟡 P5 finding moduleId values follow the canonical pN-NN format and are registered in SCAN_MODULES in data.ts
  Scenario    : Inspect each P5 module's emitted RawFinding.moduleId against the SCAN_MODULES registry in src/lib/data.ts. Run the existing scanner module registration test if one exists.
  Pass if     : Every P5 module emits findings with a moduleId matching its registered identifier (P5-01 through P5-06). Every P5 module has a SCAN_MODULES entry with display metadata.
  Fail if     : Any P5 finding has an unregistered moduleId. Any SCAN_MODULES entry is missing for a P5 module.
  Automatable : yes
  @functional

## P5-01 Cookie Consent Detection

🟡 P5-01 detects at least one CMP from the supported list (OneTrust, Cookiebot, Osano, Termly, Usercentrics, CookieYes, Didomi, Complianz, Klaro, Quantcast) when its DOM or script-domain signal is present in crawl data
  Scenario    : For each CMP vendor, construct a CrawlResult with the vendor's known DOM marker or script domain in html/jsBundleUrls. Run P5-01 and inspect findings.
  Pass if     : The module emits a signal finding (e.g. INFO severity, stating consent mechanism detected) for each fixture. The CMP vendor name is identified in the finding.
  Fail if     : Any supported CMP goes undetected when its canonical marker is present. The module emits a "no consent mechanism" finding for a site that has a CMP.
  Automatable : yes
  @functional

🟡 P5-01 emits a finding when non-essential cookies are present and no CMP signal is detected
  Scenario    : Construct a CrawlResult with multiple third-party analytics cookies (e.g. _ga, _fbp) in crawl.cookies and no CMP markers in html or jsBundleUrls. Run P5-01.
  Pass if     : The module emits a finding noting cookies that may require consent are present but no consent mechanism was detected.
  Fail if     : The module emits no finding, or the finding asserts the site violates GDPR rather than describing the observable state.
  Automatable : yes
  @functional

🟢 P5-01 does not emit a "missing CMP" finding when the site has no cookies at all
  Scenario    : Construct a CrawlResult with an empty cookies array and no third-party script domains. Run P5-01.
  Pass if     : The module emits zero findings or an INFO finding. It does not flag absence of a CMP as a problem for a site with no apparent cookie use.
  Automatable : yes
  @non-blocker

  Scenario    : Construct a CrawlResult with a complete set of data-protection headers all present. Run both P1-03 and P5-03. Compare findings for overlapping categories.
  Pass if     : P5-03 findings are categorised under P5 (moduleId P5-03) and add a privacy/data-protection framing distinct from the security framing of P1-03. No identical finding title appears from both modules. The deduplication strategy (if any) is deterministic.
  Fail if     : P5-03 re-emits findings with identical titles and severity to P1-03, causing the same issue to appear twice in the report with different module badges.
  Automatable : yes
  @functional

  Scenario    : Construct a CrawlResult with Strict-Transport-Security (max-age > 0), Content-Security-Policy, Referrer-Policy: strict-origin-when-cross-origin, Permissions-Policy (camera/mic/geolocation denied), and X-Content-Type-Options: nosniff all present.
  Pass if     : P5-03 emits zero findings or only INFO-severity informational findings.
  Automatable : yes
  @non-blocker

## P5-04 WCAG Attestation (Lighthouse a11y reuse)

🟡 P5-04 surfaces the Lighthouse accessibility score as a signal and links it to an accessibility-statement page if one is detected
  Scenario    : Provide a context argument with accessibilityScore = 72 and a CrawlResult html containing an anchor to "/accessibility-statement". Run P5-04.
  Pass if     : The finding references the Lighthouse score (72/100) as the source of the signal. The accessibility-statement link is noted. No WCAG 2.2 AA conformance claim is made.
  Fail if     : The finding claims WCAG 2.2 AA compliance at score 72. The finding omits the Lighthouse-subset caveat (not full WCAG 2.2 AA).
  Automatable : yes
  @functional

🟡 P5-04 includes a Lighthouse-subset caveat in every finding it emits
  Scenario    : Run P5-04 with accessibilityScore values of 0, 50, and 100. Inspect the explanation field of each emitted finding.
  Pass if     : Every emitted finding's explanation acknowledges that the score reflects a Lighthouse subset of WCAG checks, not a full WCAG 2.2 AA audit.
  Fail if     : Any finding omits this caveat, particularly when the score is high (near 100), where the absence could mislead a user into claiming WCAG conformance.
  Automatable : yes
  @functional

## P5-05 Third-Party Data Sharing (reuse of p1-09)

🟡 P5-05 reuses the p1-09 domain classifier to enumerate data processors without duplicating the classifier logic and without modifying p1-09 itself
  Scenario    : Inspect the P5-05 source file for its import structure. Run P5-05 and P1-09 independently on the same CrawlResult fixture containing known domains (e.g. google-analytics.com, js.stripe.com). Compare the classified domains.
  Pass if     : P5-05 imports the classifier function from p1-09 (or a shared utility it extracts) rather than rewriting it. Both modules classify the same domains identically. P1-09 test coverage is unchanged.
  Fail if     : P5-05 contains a duplicated copy of the KNOWN_DOMAINS map or classifyDomain function. Any change to P1-09 was required to make P5-05 work (P1-09 must remain unmodified).
  Automatable : yes
  @functional

🟡 P5-05 enumerates processors by category and notes whether a privacy-policy link is present (heuristic cross-check), without fetching the policy content
  Scenario    : Construct a CrawlResult with analytics and ad domains in jsBundleUrls and a privacy policy anchor in html. Run P5-05.
  Pass if     : The finding lists the detected data processors grouped by category (analytics, ads, payment, etc.). The finding notes presence or absence of a privacy policy link as a heuristic signal. No outbound request to the privacy policy URL is made.
  Fail if     : The module fetches the policy URL. The module omits the processor category breakdown. The finding claims the site does or does not disclose data sharing (a legal conclusion).
  Automatable : yes
  @functional

## Scanner Orchestrator Integration

🟡 The P5-Compliance group is added to moduleFindingCounts with the correct count when complianceScanning is enabled
  Scenario    : Run runScanner() with complianceScanning enabled against a URL that produces P5 findings. Inspect the returned moduleFindingCounts object.
  Pass if     : moduleFindingCounts['P5-Compliance'] equals the sum of findings from all six P5 modules. The key is absent when the flag is off (see flag-off tests).
  Fail if     : The count is off by any amount (e.g. only counts the last module's findings). The key appears with value 0 when the flag is off.
  Automatable : yes
  @functional

🟡 A thrown exception in any individual P5 module does not abort the entire scan
  Scenario    : Inject a synthetic throw inside the P5-03 module and run runScanner(). Observe whether the scan completes and what the result shape contains.
  Pass if     : The scan completes with all non-P5 findings and P5 findings from the non-throwing modules. A logger.error call is observed. The thrown module's findings are absent (not partial/corrupt).
  Fail if     : The exception propagates out of runComplianceModules and causes runScanner to throw, aborting the scan entirely.
  Automatable : yes
  @functional

🟢 The compliance orchestrator (compliance.ts) mirrors the structural pattern of performance.ts, accessibility.ts, and seo.ts
  Scenario    : Compare the module signatures, return types, and error handling patterns of compliance.ts against the existing group orchestrators.
  Pass if     : The file structure is consistent: a named export runComplianceModules, a ComplianceResult type, per-module calls aggregated into a combined findings array.
  Fail if     : The orchestrator uses a significantly different pattern requiring callers to handle it differently from the other group wrappers.
  Automatable : no
  @non-blocker
